        <title>Отличия Java (11-17) от Kotlin</title>
        <groups>
            <group>Kotlin</group>
        </groups>
        <short>Kotlin — это более современный, лаконичный и безопасный язык, чем Java. В Kotlin нет null-безопасности по умолчанию, что резко сокращает количество NPE. Kotlin поддерживает функциональное программирование (лямбды, функции высшего порядка) и расширения (extension functions), а также умные приведения типов. Java 17 (LTS), в свою очередь, получила значительные улучшения: текстовые блоки, switch-выражения, записи (record), классы sealed и паттерн-матчинг. Однако синтаксис остаётся более "многословным", и null-безопасность всё ещё требует явной проверки через <code>Optional</code> или аннотации.</short>
        <long>Основная философская разница: Java — зрелый, консервативный язык, ориентированный на стабильность и обратную совместимость. Kotlin — молодой, более выразительный язык, созданный для удобства разработчика и повышения продуктивности.<br/><br/>Kotlin отличается от Java рядом важных языковых конструкций:<br/>1. <b>Null Safety</b>: Типы данных по умолчанию не могут быть <code>null</code>. Для nullable типов требуется явно указывать <code>?</code> и использовать безопасный вызов (<code>?.</code>) или оператор Элвиса (<code>?:</code>). Это практически исключает <code>NullPointerException</code> на уровне языка.<br/>2. <b>Data Classes</b>: Однострочное объявление класса, автоматически генерирующее <code>toString()</code>, <code>equals()</code>, <code>hashCode()</code> и <code>copy()</code> (аналог <code>record</code> в Java 14+).<br/>3. <b>Extension Functions</b>: Возможность добавлять новые функции к существующим классам без их наследования (в Java требуется утилитный класс).<br/>4. <b>Coroutines</b> (не путать с виртуальными потоками): Легковесные потоки для асинхронного программирования, которые делают код неблокирующим и читаемым (без цепочек callback'ов). Java для этого использует <code>CompletableFuture</code> или (с 21 версией) виртуальные потоки.<br/>5. <b>Smart Casts</b>: После проверки <code>is</code> (instanceof) переменная автоматически приводится к нужному типу, без явного <code>(Type)</code>.<br/><br/>Java 17 же сделала большой шаг вперёд, добавив:<br/>- <b>Switch Expressions</b>: <code>switch</code> теперь может возвращать значение и выглядит более компактно.<br/>- <b>Records</b>: Транспортные объекты-носители данных.<br/>- <b>Sealed Classes</b>: Ограничение иерархии наследования.<br/>- <b>Text Blocks</b>: Многострочные строки.<br/><br/>Однако, Java всё ещё требует больше кода для выполнения простых задач и не имеет null-безопасности на уровне компилятора.<br/><br/>Выбор между языками часто определяется не техническими, а экосистемными причинами: Java — стандарт для корпоративных систем, Kotlin — выбор для стартапов, Android-разработки и проектов, где важна скорость разработки.</long>

        <title>Spring Framework (что такое, зачем, приемущества и недостатки)</title>
        <groups>
            <group>Spring</group>

        <short>Spring — это фреймворк для Java, который предоставляет инфраструктуру для создания корпоративных приложений. Его ядро — это DI (внедрение зависимостей) и AOP (аспектно-ориентированное программирование). Основное преимущество — слабая связанность кода, хорошая тестируемость и огромная экосистема модулей (Spring Boot, Spring Data, Spring Security). Основной минус — сложность конфигурации (хотя Spring Boot это упрощает) и некоторая магия, которая может мешать отладке.</short>
        <long>Spring Framework — это швейцарский нож для Java-разработчика. Он стандартизирует подходы к написанию Enterprise-приложений и решает проблемы, связанные с управлением зависимостями и ЖЦ объектов, позволяя разработчику сосредоточиться на бизнес-логике.<br/><br/><b>Core Components</b>:<br/>1. <b>IoC Container (DI)</b>: Самый важный компонент, реализующий инверсию управления (IoC). Созданием объектов и внедрением их зависимостей управляет контейнер (ApplicationContext). Это делает код легко тестируемым и слабосвязанным.<br/>2. <b>Spring MVC</b>: Модуль для построения веб-приложений по паттерну Model-View-Controller.<br/>3. <b>Spring AOP (Aspect-Oriented Programming)</b>: Вынесение сквозной логики (логирование, транзакции, безопасность) в отдельные модули (аспекты).<br/>4. <b>Spring Boot</b>: Экосистема авто-конфигураций, которая упрощает настройку проекта (starter'ы) и избавляет от XML-конфигураций.<br/>5. <b>Spring Data</b>: Упрощает работу с базами данных (JPA, MongoDB, Redis) через Repository-интерфейсы.<br/>6. <b>Spring Security</b>: Мощный модуль для аутентификации и авторизации.<br/><br/><b>Преимущества</b>:<br/>- <b>Слабосвязанный и тестируемый код</b> (DI позволяет легко заменять зависимости в тестах).<br/>- <b>Огромное комьюнити</b> и документация.<br/>- <b>Модульность</b> (можно использовать только нужные части).<br/>- <b>Готовые решения</b> для типичных задач (Spring Boot).<br/><br/><b>Недостатки</b>:<br/>- <b>Сложность дебага</b>: Рефлексия и прокси-объекты могут скрывать логику.<br/>- <b>"Магия"</b>: Аннотации и автоконфигурации иногда делают поведение неочевидным.<br/>- <b>Требуется время</b> на изучение (способы конфигурации).<br/>- <b>Избыточность для микро-сервисов</b>: Для простых задач может быть тяжеловесным.</long>

        <title>микросервисы/микросервисная архитектура (плюсы и минусы)</title>
        <groups>
            <group>микросервисы</group>
        </groups>
        <short>Микросервисная архитектура — это подход, где приложение разбивается на набор маленьких, независимых сервисов, каждый из которых выполняет одну бизнес-функцию. Сервисы общаются по сети (REST, gRPC, Kafka). Основные плюсы: гибкость, технологическая свобода, независимый деплой и масштабирование. Минусы: высокая операционная сложность (DevOps), сложности с распределёнными транзакциями, межсервисными тестами и сетевым общением (включая задержки и безопасность).</short>
        <long>Микросервисная архитектура — это эволюция монолитной архитектуры, которая решает проблемы масштабирования и гибкости разработки. Суть не в размере сервиса, а в его независимости.<br/><br/><b>Ключевые принципы:</b><br/>- <b>Единственная ответственность</b>: Каждый микросервис отвечает за конкретную бизнес-задачу.<br/>- <b>Слабая связанность и высокая связность</b>: Внутри сервиса логика сильно связана, но с другими сервисами — слабо.<br/>- <b>Технологическая свобода</b>: Каждая команда может выбрать свой стек.<br/>- <b>Независимый деплой</b>: Обновление одного сервиса не требует перезапуска всей системы.<br/>- <b>Доменно-ориентированная архитектура</b>: Группировка микросервисов по доменным областям (DDD).<br/><br/><b>Плюсы</b>:<br/>- <b>Масштабируемость</b>: Можно масштабировать только проблемный сервис.<br/>- <b>Отказоустойчивость</b>: Сбой одного сервиса не падает всей системы (если реализованы паттерны resilience).<br/>- <b>Независимый жизненный цикл</b>: Не ждём релиза других команд.<br/>- <b>Ускорение разработки</b>: Команды работают параллельно над разными сервисами.<br/><br/><b>Минусы</b>:<br/>- <b>Сетевая сложность</b>: latency, безопасность, сериализация/десериализация.<br/>- <b>Distributed Transactions</b>: Требуются паттерны Saga или двухфазная фиксация.<br/>- <b>DevOps нагрузка</b>: Требуется зрелый CI/CD, контейнеризация (Docker), оркестрация (Kubernetes), логирование и мониторинг.<br/>- <b>Сложность end-to-end тестирования</b>: Требует запуска всей системы или контрактных тестов.<br/>- <b>Data Consistency</b>: Данные часто дублируются в разных сервисах, и вопрос "чьи это данные" становится нетривиальным.</long>
    </question>

        <title>PostgreSQL (плюсы и минусы по сравнению с другими СУБД)</title>
        <groups>
            <group>PostgreSQL</group>
        </groups>
        <short>PostgreSQL — объектно-реляционная СУБД, наиболее полная и функциональная из Open Source баз. Главные преимущества: расширяемость (добавление своих типов данных), поддержка JSON/JSONB (работа с документными данными) и строгое соблюдение ACID. Основные недостатки: значительное потребление памяти (размер индексов) и сложность горизонтального масштабирования по сравнению с NoSQL-решениями (требуется внешний шардинг).</short>
        <long>PostgreSQL — мощная и зрелая СУБД, которая часто выбирается как стандарт для сложных, транзакционных систем с высокой нагрузкой.<br/><br/><b>Основные особенности и преимущества:</b><br/>1. <b>Расширяемость</b>: PostgreSQL позволяет создавать свои типы данных (пользовательские домены), операторы и функции на разных языках (PL/pgSQL, PL/Python, PL/Java). Это делает её платформой, а не просто хранилищем.<br/>2. <b>Поддержка JSON/JSONB</b>: Это мощная альтернатива NoSQL. JSONB позволяет строить индексы и эффективно работать с частично структурированными данными внутри реляционной модели.<br/>3. <b>ACID и MVCC</b>: Полное соблюдение ACID и реализация многоверсионности для параллельной работы без блокировок.<br/>4. <b>Отказоустойчивость</b>: Встроенные механизмы репликации (Streaming Replication), горячие резервные копии и восстановление до точки времени.<br/>5. <b>Индексы</b>: Поддерживает B-tree, Hash, GiST, SP-GiST, GIN и BRIN — огромный арсенал для производительности.<br/><br/><b>Минусы и ограничения:</b><br/>- <b>Горизонтальное масштабирование</b>: Не встроено. Требуется внешний шардинг (Citus или ручная логика), в отличие от Cassandra или MongoDB.<br/>- <b>Производительность в некоторых сценариях</b>: Может уступать узкоспециализированным СУБД для аналитики (ClickHouse).<br/>- <b>Потребление памяти</b>: Для поддержки больших индексов требуется достаточно ресурсов.<br/>- <b>Сложность настройки</b>: Требует глубоких знаний администратора для тюнинга производительности.</long>

        <title>Cassandra (особенности, плюсы и минусы по сравнению с другими noSQL)</title>
        <groups>
            <group>Cassandra</group>
        </groups>
        <short>Cassandra — распределённая NoSQL СУБД, спроектированная для линейной масштабируемости и высокой доступности в распределённых системах. Хранит данные в таблицах по принципу ключ-значение. Идеальна для больших объёмов записей (write-heavy) и временных рядов. Минусы: нет ACID-транзакций, агрегации требуют тщательного проектирования и негибкая модель запросов (запросы только по ключам).</short>
        <long>Apache Cassandra — децентрализованная база данных, которая решает проблему высокой доступности и масштабируемости в системах, критических к отказам. Создана по образу Amazon DynamoDB и Bigtable.<br/><br/><b>Ключевые особенности:</b><br/>1. <b>Децентрализация</b>: Все узлы равны, нет единой точки отказа.<br/>2. <b>Tunable Consistency</b>: Можно настроить уровень консистентности для каждой операции (от быстрой, но слабой, до строгой, но медленной).<br/>3. <b>Горизонтальное масштабирование</b>: Добавление узлов линейно увеличивает пропускную способность.<br/>4. <b>Data Model</b>: Хранилище широких столбцов (Wide-column store). Моделируется вокруг запросов (data modelling by queries).<br/>5. <b>Композитные ключи</b>: Первичный ключ состоит из partition key (определяет распределение) и clustering keys (определяет порядок внутри узла).<br/><br/><b>Плюсы</b>:<br/>- <b>Высокая доступность</b>: Отказоустойчивость в распределённых средах (например, облака).<br/>- <b>Write throughput</b>: Очень высокая скорость записи (LSM-деревья).<br/>- <b>Регулируемая консистентность</b>: Гибкость для разных сценариев.<br/>- <b>Распределённый характер</b>: Хорошо подходит для географически-распределённых систем.<br/><br/><b>Минусы</b>:<br/>- <b>Отсутствие ACID</b>.<br/>- <b>Ограниченные запросы</b>: Нельзя запрашивать данные по полям, не входящим в первичный ключ (без использования вторичных индексов, которые имеют ограничения).<br/>- <b>CQL (Cassandra Query Language)</b>: Похож на SQL, но значительно беднее. Нет JOIN, GROUP BY или подзапросов (их нужно моделировать на стороне приложения).<br/>- <b>Сложное управление</b>: Обслуживание кластера (rebalancing, repartitioning, repairs) требует навыков.<br/>- <b>Читает медленнее</b>, чем записывает (из-за LSM-структуры).</long>

        <title>Kafka (плюсы и минусы по сравнению с RabitMQ)</title>
        <groups>
            <group>Kafka</group>
        </groups>
        <short>Kafka — распределённая платформа для потоковой передачи событий. Это не классический брокер сообщений, а распределённый лог (commit log). Отличия от RabbitMQ: огромная пропускная способность (млн событий/сек), долговременное хранение данных на диске и возможность перечитывать сообщения (replay). Минусы: более сложная настройка, нет встроенных механизмов маршрутизации (только по топикам), нет приоритетов сообщений и Dead Letter Queues.</short>
        <long>Apache Kafka — это де-факто стандарт для построения систем потоковой обработки данных в реальном времени. В основе Kafka лежит модель "распределённого журнала".<br/><br/><b>Архитектура:</b><br/>- <b>Broker</b>: Сервер, хранящий данные.<br/>- <b>Topic</b>: Логическая единица группировки данных. Топик разбит на партиции.<br/>- <b>Partition</b>: Физический блок данных, строго упорядоченный.<br/>- <b>Producer</b>: Публикует сообщения в топик.<br/>- <b>Consumer</b>: Читает сообщения из топика, управляя offset'ом.<br/>- <b>Consumer Group</b>: Группа потребителей, делящая партиции между собой.<br/><br/><b>Ключевые отличия от RabbitMQ (важнейший блок для интервью):</b><br/>1. <b>Модель получения</b>: RabbitMQ — Push (брокер отправляет данные). Kafka — Pull (потребитель сам вытягивает данные).<br/>2. <b>Хранение данных</b>: RabbitMQ удаляет сообщение после ack. Kafka хранит всё заданное время (retention), позволяя replay.<br/>3. <b>Маршрутизация</b>: RabbitMQ — гибкая (Exchanges, Bindings). Kafka — только по топикам.<br/>4. <b>Производительность</b>: Kafka — миллионы сообщений/сек, RabbitMQ — десятки тысяч.<br/><br/><b>Плюсы</b>:<br/>- <b>Высокая пропускная способность</b> (идеально для стриминга логов, метрик, событий).<br/>- <b>Устойчивость к сбоям</b> (репликация партиций).<br/>- <b>История событий</b>.<br/>- <b>Горизонтальное масштабирование</b> через добавление партиций.<br/><br/><b>Минусы</b>:<br/>- <b>Задержка</b> может быть выше, чем в RabbitMQ (из-за диссинхронизации с диском).<br/>- <b>Нет встроенных DLQ</b>.<br/>- <b>Нет приоритетов сообщений</b>.<br/>- <b>Управление consumer group</b> (rebalance) может быть сложным.</long>

        <groups>

        <short>Docker — технология контейнеризации, которая упаковывает приложение и все его зависимости (библиотеки, конфиги) в изолированный контейнер. Главное преимущество — единообразие окружения от разработки до продакшена ("работает у меня" больше не проблема). Минусы: дополнительный слой абстракции и потребление ресурсов (хотя значительно меньше, чем у виртуальных машин).</short>
        <long>Docker — это платформа для запуска приложений в изолированных средах (контейнерах). Контейнер похож на виртуальную машину, но использует ядро хост-системы (Linux), что делает его лёгким и быстрым.<br/><br/><b>Как работает:</b><br/>- В основе лежат возможности ядра Linux: namespaces (изоляция процессов, сетей, пользователей) и cgroups (ограничение ресурсов CPU/RAM).<br/>- <b>Docker Engine</b>: Это серверный процесс (daemon), который управляет контейнерами, образами и сетями.<br/>- <b>Docker Image</b>: Снимок файловой системы, содержащей приложение, его зависимости, переменные окружения и инструкции по запуску.<br/>- <b>Dockerfile</b>: Текстовый файл с инструкциями по сборке образа.<br/>- <b>Docker Registry</b>: Хранилище образов (Docker Hub, частные registry).<br/><br/><b>Преимущества:</b><br/>- <b>Переносимость</b>: Контейнер работает одинаково на любой ОС с Docker Engine.<br/>- <b>Изоляция</b>: Контейнеры изолированы друг от друга и от хоста.<br/>- <b>Управление версиями</b>: Образы можно тегировать, отслеживая версии приложений.<br/>- <b>Масштабирование</b>: Легко масштабировать количество контейнеров (вручную или через Kubernetes).<br/>- <b>Упрощение CI/CD</b>: Образы собираются один раз и деплоятся на любые окружения.<br/><br/><b>Минусы:</b><br/>- <b>Новая сложность</b>: Требуется изучение новых понятий.<br/>- <b>Безопасность</b>: Контейнеры делят ядро с хостом.<br/>- <b>Network Overhead</b>: Небольшие потери производительности.<br/>- <b>Управление данными</b>: Данные в контейнере не сохраняются (это решается томами — volumes).</long>

        <groups>
            <group>Kubernetes</group>

        <short>Kubernetes (K8s) — это система оркестрации контейнеров, т.е. автоматизированного управления контейнерами на масштабе кластера. Это не просто "Docker в большом масштабе".         <short>Паттерны проектирования — это готовые решения для часто встречающихся проблем в ООП. Делятся на порождающие (создание объектов, Singleton), структурные (построение иерархий, Adapter, Decorator) и поведенческие (взаимодействие, Observer). Изоляция кода и возможность его масштабирования — главный плюс. Минус — усложнение кода и риск "перепроектирования" (overengineering), когда вместо простого решения выбирают паттерн.</short>
        <long>Паттерны проектирования — это проверенные временем решения для типовых проблем объектно-ориентированного проектирования. Они делятся на три категории.<br/><br/>1. <b>Порождающие паттерны (Creational)</b><br/>   - <b>Singleton</b>: Гарантирует единственный экземпляр класса.<br/>   - <b>Factory Method</b>: Создание объектов через фабричные методы.<br/>   - <b>Abstract Factory</b>: Создание семейств связанных объектов.<br/>   - <b>Builder</b>: Пошаговое создание сложных объектов.<br/>   - <b>Prototype</b>: Клонирование объектов.<br/><br/>2. <b>Структурные паттерны (Structural)</b><br/>   - <b>Adapter</b>: Преобразует интерфейс одного класса в интерфейс, ожидаемый клиентом.<br/>   - <b>Decorator</b>: Динамическое добавление функциональности.<br/>   - <b>Proxy</b>: Заместитель для контроля доступа.<br/>   - <b>Composite</b>: Древовидные структуры (Folder-File).<br/>   - <b>Facade</b>: Упрощенный интерфейс для сложной подсистемы.<br/><br/>3. <b>Поведенческие паттерны (Behavioral)</b><br/>   - <b>Observer</b>: Уведомление объектов об изменении состояния (Event Listeners).<br/>   - <b>Strategy</b>: Выбор алгоритма на лету.<br/>   - <b>State</b>: Изменение поведения объекта при изменении состояния.<br/>   - <b>Command</b>: Инкапсуляция запроса в объект (для очередей).<br/>   - <b>Template Method</b>: Определение скелета алгоритма.<br/><br/><b>Сильные стороны паттернов</b>:<br/>- <b>Универсальность</b>: Решения, понятные всем разработчикам.<br/>- <b>Упрощение рефакторинга</b>: Благодаря слабой связанности.<br/>- <b>Повышение качества</b>: Проверенные решения уменьшают количество ошибок.<br/><br/><b>Слабые стороны</b>:<br/>- <b>Усложнение кода</b>: Паттерны требуют дополнительных классов.<br/>- <b>Риск overengineering</b>: Применение "ради применения".<br/>- <b>Не всегда эффективны</b>: В простых задачах работают медленнее прямолинейных решений.</long>
